# 謝秀玟｜面試自我介紹講稿

建議總長度：約 5 分鐘。

## 第 1 頁：封面

建議時間：30 秒

您好，我是謝秀玟，過去三年在晶心科技擔任軟體工程師，主要負責 Eclipse IDE 相關工具與除錯功能，也處理 CI/CD 測試流程改善。

如果用一句話介紹我的工作特質，我會說我比較擅長在複雜系統裡找出問題根因，再把不穩定或重複性的流程，整理成可維護、可自動化的工程解法。

## 第 2 頁：大綱

建議時間：15 秒

今天我會先分享我在晶心科技的核心經驗，包含 RISC-V IDE 工具整合、幾個重要的功能開發與視圖重構，然後聯兩個比較有代表性的成果，最後談近期的 AI 與全端方向。

## 第 3 頁：RISC-V IDE 工具整合

建議時間：45 秒

我過去比較核心的工作，是把其他團隊開發的 RISC-V 晶片工具整合到 Eclipse IDE 裡，讓嵌入式工程師可以直接在 IDE 內檢視暫存器、做多核心除錯和效能分析。

這類工作不只是把畫面做出來，還需要理解底層工具鏈、GDB、Eclipse CDT，以及不同作業系統環境的限制。對我來說，這段經驗訓練的是如何把底層能力包成工程師實際用得起來的產品功能。

## 第 4 頁：功能開發與視圖重構

建議時間：40 秒

在這個專案裡，我也負責了幾個重要功能的開發和重構。像是 AutoOpTune 視圖，用 Web View 呈現自動調優的分析結果，過程中需要在已停止維護的 CentOS 7.4 環境上讓瀏覽器核心正常運作，花了不少時間處理相容性。

另外也重構了 Vector Register 和 MMU 視圖，把原本耦合在 UI 框架的邏輯拆出來，配置改用 JSON 外部化。還有 Multi-Project 架構，讓 IDE 可以管理多個子專案的建立和建置，支援更複雜的 multi-core debug 場景。

## 第 5 頁：GUI 自動化測試重構

建議時間：55 秒

其中一個我比較有代表性的例子，是 GUI 自動化測試重構。當時 QF-Test 在 CI 裡常出現 component not found，導致需要人工重跑。

我沒有只針對失敗腳本一直補 wait，而是從 runlog、錯誤訊息和執行時序一路往下追。最後確認問題跟 Daemon 模式下 Java Swing/AWT 的渲染時序不同步有關。

後來我調整測試的執行架構，讓測試能在 Jenkins Agent 本地更穩定地執行。調整後，這類錯誤從原本頻繁出現降到基本為零，也減少了人工重跑的成本。

## 第 6 頁：規格文件轉換工具

建議時間：55 秒

另一個例子是規格文件轉換工具。當時工具需要大量 CSR 和 bitfield 的描述資料，但這些定義散落在 SPA 規格文件中，如果人工轉換，大概要花 5 天，而且很容易出錯。

我後來開發了一個自動轉換工具，解析文件中的暫存器名稱、位址和 bitfield 定義，並自動輸出 JSON、GDB XML 和 C header。

這個工具把原本 5 天的手工作業縮短到幾分鐘。更重要的是，後續規格更新時，只要重跑腳本就能產出新版本，降低了人工錯誤和版本維護成本。

## 第 7 頁：近期 AI Agent 與全端 PWA

建議時間：45 秒

近期我也在個人 side project 裡嘗試把 AI Agent 工作流整合到全端 PWA。這個專案是 Travel Dashboard，主要做旅遊資料管理、文件匯入、離線操作和同步。

技術上，我使用 React、Vite 和 Firebase，並整合多模型 API、Function Calling、文件解析與座標自動補齊。對我來說，AI 的重點不是追流行，而是把重複整理和資料轉換的工作交給工具，讓工程師把時間放在架構決策和問題分析上。

## 第 8 頁：謝謝

建議時間：10 秒

謝謝。
